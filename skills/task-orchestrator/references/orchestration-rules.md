# 编排规则

## 任务拆解流程

### 步骤1：理解需求
- 分析用户意图，提取关键需求
- 识别隐含需求（用户没说但可能需要的前置/后置步骤）
- 输出：需求摘要 + 关键词

### 步骤2：拆解为子任务
- 将复杂任务拆解为独立子任务
- 每个子任务应该是单个工具可以完成的
- 标注依赖关系（哪些子任务可以并行，哪些必须串行）

### 步骤3：分配工具
- 对每个子任务，按优先级匹配工具：MCP → Skill → CLI → 直接编码
- 为每个工具撰写合适的提示词（清晰、具体、包含上下文）

### 步骤4：定义降级方案
- 每个子任务至少有一个 Plan B
- Plan B 可以是不同工具、不同方法、或简化版方案

### 步骤5：生成计划并确认
- 展示完整计划给用户
- 等待确认后自动执行（不再逐步确认）

## 并行 vs 串行决策

并行条件：
- 子任务之间没有数据依赖
- 不修改相同的文件
- 不需要前一步的输出作为输入

串行条件：
- 子任务 B 依赖子任务 A 的输出
- 修改相同的文件需要合并
- 需要前一步的结果做决策

## 提示词构建规则

对每个子任务分配 agent 时，构建提示词必须包含：
1. 当前任务在整个流程中的位置
2. 具体的输入/输出要求
3. 上下文信息（之前的步骤结果）
4. 降级方案提示（如果失败，尝试什么替代方案）
